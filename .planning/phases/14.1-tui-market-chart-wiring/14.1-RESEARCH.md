# Phase 14.1: TUI Market Panel + Terminal Chart — Research

## Existing Shape

Phase 14 deliberately shipped `MarketPanel` as a placeholder. The application already has:

- selected ticker state in `TuiState.setup.ticker`
- slash command palette and ticker picker patterns
- local analysis market-data fetching with deterministic fallback
- pure indicator functions used by the Technical Agent

This means Phase 14.1 should be a wiring/product phase, not a new architecture phase.

## Terminal Chart Feasibility

A Streamlit/Plotly-grade interactive chart is not realistic inside a plain terminal without adding substantial dependencies or launching a browser. However, terminal-native charting is feasible and useful:

- price sparkline using Unicode block characters or Braille-like density
- volume mini-bars below price
- latest OHLCV row and daily change
- indicator summary table: RSI, MACD signal, SMA20/SMA50, EMA values
- optional selected ticker screen navigated by `/chart` and `/chart TICKER`

This is sufficient for a local-first TUI cockpit and keeps the app fast/offline-friendly.

## Recommended Storage/Service Shape

Add a small local market snapshot service independent of LLM workflow:

```text
src/apex/services/market_snapshot.py
```

Suggested return shape:

```python
{
    "ticker": "AAPL",
    "bars": [...],
    "latest": {"open": ..., "high": ..., "low": ..., "close": ..., "volume": ...},
    "indicators": {"rsi": ..., "macd": ..., "sma20": ..., "ema50": ...},
    "source": "live" | "fallback",
}
```

The service should never call an LLM. It should use existing OHLCV fetch and deterministic fallback so `/chart` remains cheap.
